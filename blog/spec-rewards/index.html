Specifications Are Decision Boundaries

May 2026 · Prannay Hebbar

A model that “makes the door blue” can also silently rewrite three other paths in your SVG. Pixel metrics won't catch it. RLHF won't catch it. But fifty years of program-synthesis ideas, mutation testing, counterexample-guided induction, version-space algebras, rearrange neatly into a way to train agents that know what not to change.

This is a short essay on the load-bearing idea behind a paper I am working on called VectorSpecGym. The paper trains a 7B open model with RL on SVG editing. The idea generalises far past SVGs.

01The silent rewrite

Consider the instruction: “make only the door blue.” Below are three outputs. The raster looks identical. The XML does not.

This is the visual analogue of a unit test that passes while corrupting an invariant nobody wrote a test for. Surface metrics reward the picture; specifications care about the diff.

02Specification leakage

This is one failure mode wearing many costumes. The unit test passes; the migration corrupted a column. The lint is clean; the refactor renamed a public symbol. The render looks right; the SVG tree is unrecognisable. In every case the agent satisfied what was measured, not what was meant.

Call this specification leakage. It is the bottleneck on every agent domain with structured outputs, code, UI trees, CAD scripts, robot policies. Pixels are simply the easiest place to see it bite.

03An old idea: counterexample-guided induction

The trick of grading a candidate against a structured set of accept/reject examples is not new. It's the central loop of program synthesis since the 2000s.

Armando Solar-Lezama's Sketch system [Solar-Lezama 2006] introduced what is now the dominant pattern in the field: counterexample-guided inductive synthesis (CEGIS). A synthesiser proposes a candidate program. A verifier, typically an SMT solver, checks it against a logical spec. If the verifier finds a violation, it returns a concrete counterexample input. The synthesiser refines on the expanded example set. Loop until the verifier signs off.

Sketch [2006], BRAHMA [Gulwani 2011], Rosette [Torlak & Bodík 2013], Synthesis-style superoptimisers, all CEGIS underneath. The verifier is the only sharp tool in the toolkit.

What we are doing has the same shape, with two substitutions. The verifier becomes a mutant generator: it does not solve constraints, it pre-computes a family of accept/reject artifacts per task. The synthesiser becomes a neural policy: it does not search over programs symbolically, it samples from a distribution and gets gradient updates. Negative mutants are exactly what CEGIS calls counterexamples, concrete artifacts the policy must learn not to produce. Positive mutants are witnesses, the synthesis literature's name for “this also satisfies the spec.”

The reason this works at all is the same reason CEGIS works: a small set of well-chosen accept/reject examples nails down a specification much more precisely than any single reference output ever could.

04Where the mutants come from

CEGIS needs a verifier. Ours comes from a fifty-year-old idea on the test-quality side of the same field: mutation testing [Lipton 1971; DeMillo, Lipton, Sayward 1978]. Originally a tool to grade test suites, programmatically perturb a function, then ask whether the tests notice, we flip its role entirely.

Use mutants as labels, not as adversaries. For every editing task, the same code that generates the task also generates two structured sets of nearby artifacts.

Top (M⁺): different valid blues. The door is the door; only the colour varies. Bottom (M⁻): a window has been deleted, the roof has shifted, the door has moved, or the wrong object was recoloured. The specification is the line drawn between the two sets.

You now have ground truth without humans, without a learned judge, without a single canonical reference output. The mutants are produced by the same family-specific code that defines the task. They are free.

05Specifications as decision boundaries

A specification is the boundary that separates M⁺ from M⁻, not a reference answer, but a region of program space defined by what it is closer to than to its alternatives.

This is the load-bearing idea. The type-theoretic name for it already exists: a refinement type [Freeman & Pfenning 1991; Liquid Haskell] is a base type sharpened by a predicate that carves out a subset of inhabitants. { door : Colour | door = blue ∧ ∀o ≠ door. o = o₀ } is the spec. The mutant lattice is a sampled view of that subset and its complement. Mutation-consistency RL is, in this view, learning to produce refinement-typed outputs without ever writing the refinement down as a logical formula.

06Version spaces, or why one example isn't enough

When Sumit Gulwani's FlashFill [Gulwani 2011] shipped inside Excel, it solved a deeper problem than its UI suggested: given one or two input-output examples, find a program that generalises the way the user actually meant. The mechanism was a version-space algebra, a compact representation of every program in some DSL consistent with the examples seen so far.

What FlashFill teaches is structural: one example is almost always ambiguous. The instruction “door = blue” alone is consistent with at least the following programs:

Figure 04 · Version space for “door = blue”, every program shown is consistent with the prompt alone
prompt
“door = blue”
π₁door.fill ← #3a7ad9 (one specific blue)
π₂door.fill ← any blue from BLUES
π₃∀ d ∈ doors: d.fill ← blue
π₄door.fill ← blue; door.stroke ← blue
π₅delete door; insert new blue door at same coords
π₆tint entire scene with blue overlay
Negative mutants kill π₅ and π₆ (preservation violations) and π₄ (stroke change). What survives, π₁, π₂, π₃, is the spec.

The negative-mutant set is the disambiguator. It carves the version space down to the programs the instruction actually meant. Modern PBE systems do this with example expansion; we do it with one shot of counterexamples per task. Same primitive, different optimiser.

07Why SFT can't learn this

Supervised fine-tuning is imitation: here is the right output, copy it. To teach “leave everything you weren't asked about alone,” you would need a label for every non-change. There are infinitely many of those. The space of things to preserve is unboundedly larger than the space of things to edit, and SFT scales with example count.

RL with a counterfactual reward sidesteps the asymmetry. The reward says “you lost points for moving anything I didn't mention,” and gradient descent does the rest. Preservation is the structurally hard part of editing, and it is precisely the part SFT is incapable of cheap supervision for.

08The copy trap

Our first reward was naive: +1 for matching any positive mutant. Within a few hundred GRPO steps, the policy converged on something technically optimal and obviously wrong. It learned to copy a positive mutant verbatim, ignoring the prompt entirely. Maximum reward, zero understanding. A textbook reward hack.

Figure 05 · Reward-only-attractive collapses to a single point

Naive reward (M⁺ only)

policy → copies a single M⁺

Mutation-consistency reward

policy → spans the spec region
Each dot is one rollout from the trained policy after 1k GRPO steps. Left: pure attraction collapses every rollout to a single copied artifact. Right: adding repulsion to M⁻ and a preservation gate spreads rollouts across the valid region, which is exactly what the spec permits.

The fix is the geometry of the spec itself: subtract similarity to negative mutants (repulsion), and gate on a preservation term that requires non-referenced objects to remain bit-equivalent. Any reward that is only attractive is gameable. You need repulsion. This is Goodhart's law in its ML form, and it shows up the moment your reward stops being a comparison.

09GRPO matches the geometry

We train with GRPO [Shao et al. 2024, DeepSeekMath], group-relative policy optimisation. Sample K=8 rollouts per prompt, score each, normalise advantages within the group, reinforce the above-average ones, push away the below-average ones. No value function. No critic. Just relative ranking inside a group.

The match is not accidental. The reward is already group-relative, better than negatives, worse than positives. An algorithm that ranks within groups is the geometrically natural choice. PPO with a critic would be wasted machinery, predicting a baseline we can read directly off the mutant set.

10The lineage on one page

To place this in context: the table below is roughly the sequence of ideas this paper rearranges. None of the pieces are new. The arrangement is.

2026
Mutation-consistency RL (this work)

Mutators replace solvers; policy gradients replace candidate search; the spec is implicit in the lattice.

The same primitive, “a small set of structured examples grounds a specification”, applied at every level of the stack, from test quality to type theory to optimiser design.

11The result that matters

Prompting matches pictures. SFT matches pictures. Only RL with counterfactual rewards matches the spec. The gap is largest exactly where you'd expect, on the Unintended Change Rate, the fraction of unmentioned objects that drift, because that's the metric the other two regimes were never trained to optimise.

What is more telling than the headline number is where SFT fails: on the long tail of small, plausible, locally-invisible mutations. A minor recolor of an unmentioned object. Path data drifting under reformatting. An ID renamed in a way that breaks downstream tooling. Each of these would slip past a human reviewer glancing at the render. Mutation rewards catch them because the output measurably drifts toward a known M⁻.

13The recipe

If you want to train an agent that respects a specification, don't show it the right answer, show it a small set of right answers and a small set of plausibly-wrong answers, and let RL learn the boundary. The right answer is rarely unique; the boundary almost always is.

Two predictions I would be willing to defend in print. First, agent benchmarks will move from “one reference output” to “set of acceptable outputs plus set of forbidden ones,” because that is how you actually measure preservation. Second, RL on agents will look less like RLHF, learned reward models that inherit the same surface-metric hacks they were meant to fix, and more like this: executable counterfactual oracles, written once per task family, free to call a million times during training.

The pivot is philosophical, not algorithmic. Once you stop thinking of the spec as an answer key and start thinking of it as a boundary, almost every interesting agent-training problem rearranges itself around the same ancient idea: that meaning is operationalised by what you can distinguish from it.
